# MediaDMS — инструкция по работе с репозиторием

Этот документ — пошаговая инструкция для команды, которая только начинает работать с Git.

## 1) Правила по веткам

- `main` — только релизы.
- Прямой push в `main` запрещён всем, кроме **Андрея**.
- `dev` — основная рабочая ветка проекта.
- Прямой push в `dev` запрещён всем.
- Любые изменения делаются только через отдельную ветку и Merge Request (MR) в `dev`.

## 2) Общий рабочий процесс

1. Обновить локальную `dev`.
2. Создать новую ветку от `dev`.
3. Внести изменения в новой ветке.
4. Сделать осмысленный коммит.
5. Запушить ветку в удалённый репозиторий.
6. Создать MR в `dev`.
7. Назначить ревьюверов: **Андрей**, **Илья** и (по необходимости) **Кирилл**.
8. После создания MR пингануть всех ревьюверов в Telegram-чате.

## 3) Полезные команды Git

> Ниже команды, которые нужно выполнять в терминале.

### 3.1 Клонирование репозитория

```bash
git clone <URL_репозитория>
cd MediaDMS
```

### 3.2 Проверить, в какой вы ветке

```bash
git branch
```

Текущая ветка будет помечена `*`.

### 3.3 Получить последние изменения

```bash
git checkout dev
git pull origin dev
```

### 3.4 Создать свою ветку от `dev`

Название ветки делайте понятным, например:
- `feature/add-login-form`
- `fix/header-bug`
- `docs/update-readme`

Команда:

```bash
git checkout -b feature/короткое-описание
```

### 3.5 Внести изменения и подготовить коммит

Проверить, какие файлы изменены:

```bash
git status
```

Добавить все изменения в коммит:

```bash
git add .
```

Сделать коммит:

```bash
git commit -m "Коротко и по делу: что изменено"
```

> Коммиты можно писать на русском языке, главное — чтобы сообщение отражало суть изменений.

### 3.6 Отправить ветку в удалённый репозиторий

```bash
git push -u origin feature/короткое-описание
```

## 4) Как создать Merge Request (MR)

После `git push` обычно в терминале появляется ссылка на создание MR.
Если ссылки нет:

1. Откройте репозиторий в браузере.
2. Перейдите в раздел **Merge Requests**.
3. Нажмите **New Merge Request**.
4. Выберите:
   - **Source branch**: ваша ветка (`feature/...`)
   - **Target branch**: `dev`
5. Заполните:
   - **Title** — коротко и по сути (можно на русском)
   - **Description** — что сделано, что проверить, важные нюансы
6. В поле ревьюверов укажите:
   - **Андрей**
   - **Илья**
   - **Кирилл** (если нужно)
7. Создайте MR.

> Название и описание MR можно писать на русском, но они должны чётко объяснять суть изменений.

## 5) Что делать после создания MR

- Сразу отправьте сообщение в Telegram-чат и пинганите ревьюверов:
  - Андрей
  - Илья
  - Кирилл (если назначен)
- Дождитесь ревью и комментариев.
- При необходимости внесите правки в **ту же ветку** и запушьте повторно — MR обновится автоматически.

## 6) Частые ошибки (и как не допускать)

- ❌ Пушить напрямую в `dev` или `main`.
- ❌ Создавать ветку не от `dev`.
- ❌ Делать неинформативные сообщения коммитов (`fix`, `upd`, `123`).
- ❌ Забывать назначать ревьюверов и писать в Telegram.

✅ Всегда работаем через ветку → MR в `dev` → ревью → merge.
